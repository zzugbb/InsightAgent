# InsightAgent 实时计划

## 当前快照

- W1-W4 已完成并收口：SSE、Trace、Memory、RAG、Token/Cost、任务详情与导出、基础工作台闭环已具备。
- 阶段 5 已完成基础产品化：JWT + refresh、PostgreSQL 单后端、`RBAC-lite`、`rag-rbac-lite`、任务取消/超时、running task 恢复、usage dashboard、审计事件扩展、主链路 e2e 与 CI tooling。
- 当前活跃主线已经从 `tool-runtime-productionization` 文档收口切换到真实工具执行与 registry 治理产品化。

## 背景与目标

- 产品定位：可观测、可回放、可解释、可治理的 Agent 平台，而不只是聊天壳或单次 demo。
- 当前开发目标：在不破坏外部 SSE / trace / export / e2e 契约的前提下，把默认工具去 mock 化、真实工具接入与 registry/provider/source 治理继续做成可持续主干。
- 文档定位：本计划文件负责保留长期稳定约定与当前主线判断，不负责保存每日流水账。

## 当前固定约定

- 默认运行策略：provider/model/api_key 完整时自动走 `remote`，否则回退 canonical `mock`。
- 当前优先级：真实工具执行本体 > registry 治理产品化 > 单机并发治理 > 更细粒度 RAG 治理。
- 当前不优先：继续维护 archived runtime spec、扩写历史 progress_note、为了内部重构随意改外部契约。
- 当前测试策略：保持先补 focused failing test 再改实现；若环境阻塞，明确阻塞点，不拿旧结果冒充新结果。

## 架构与数据分工

- `PostgreSQL`：业务主存储，保存用户、会话、消息、任务、trace、usage、设置、审计。
- `Chroma Memory`：会话级语义记忆，collection 为 `memory_{session_id}`。
- `Chroma RAG`：知识库级检索，collection 为 `kb_{user_hash}_{knowledge_base_id}`。
- 前端职责：消费 SSE 与 trace，承接 workbench、task detail、settings、usage、audit、knowledge base governance 等工作台能力。

## 执行原则

1. 先主链路后边角：优先保证设置/鉴权、任务提交、Agent loop、SSE、trace、导出与回放主链路稳定。
2. 先契约后展示：外部结构尽量稳定，优先在内部 helper/runtime/display 语义上收口。
3. 先真实化后扩展：优先把已有 planned tools 的执行本体、preview/output/summary、observation、export 做真实，不优先铺更多 planner 兼容壳。
4. 先单机可靠后平台化：当前优先单机队列化、并发治理与真实工具可靠性，不提前引入分布式复杂度。

328. HTTP JSON response body adapter 的 body 属性错误诊断也已收紧：当显式 `.content` / `.body` / `.data` / `.text` body 属性是坏类型，而泛型 `__iter__` 只是空 metadata iterator 时，runner 会保留 body type 诊断，不再被空 iterable 降级成 `empty JSON response`。
329. HTTP JSON response body adapter 的 `read()` body 错误诊断也已收紧：当显式 `read()` 返回坏 body 类型，而泛型 `__iter__` 只是空 metadata iterator 时，runner 会保留 reader body type 诊断，不再被空 iterable 降级成 `empty JSON response`；空 `read()` / 空 chunked read 的后续 fallback 语义保持不变。
330. HTTP JSON response body adapter 的 `read()` 调用形态 fallback 也已补齐：当无参 `read()` 与 `read(amt)` 都只是 SDK 方法签名不适配，但 `.content` 等后续 source 明确为空时，runner 会保持 `empty JSON response`，不会把调用形态 TypeError 误报成真实 body type `transport error`。
331. HTTP JSON response body adapter 的 `json()` 调用形态 fallback 也已补齐：当 `json()` 只是 SDK 方法签名不适配，但 `.text` 等后续 source 明确为空时，runner 会保持 `empty JSON response`，不会把调用形态 TypeError 误报成 parsed-body `transport error`；真实 parsed body 类型错误仍保持 fatal 诊断。
332. HTTP JSON response body adapter 的 typed parsed body dump 方法形态 fallback 也已补齐：当根层或嵌套 typed payload 暴露的 `model_dump()` 只是 SDK/Pydantic 方法签名不适配时，runner 会继续尝试 `dict()` / `to_dict()` 并复用 response_path/result_fields 映射；真正不可序列化对象仍保留 JSON serializable 诊断，不会被误吞成假成功。
333. HTTP JSON response body adapter 的 typed dump-json 错误诊断也已收紧：`model_dump_json()` / `to_json()` / typed `.json()` 只有 TypeError 调用形态不适配时才会继续 fallback；真实 serializer 运行时异常会保留 `response json body ... failed` 诊断，即使后续 `to_dict()` 可用也不会把上游/SDK 序列化失败掩盖成成功输出。
334. HTTP JSON response body adapter 的 typed dump-json TypeError 诊断也已细化：runner 会先通过 callable signature 判断零参 dump 调用是否可用，只有签名确实不匹配时才把 TypeError 视作 fallback 信号；签名可匹配后 serializer 内部抛出的 TypeError 会保留为真实上游/SDK dump 失败，继续进入 trace/export 错误诊断。
335. HTTP JSON typed dump callable signature metadata fallback 已一次性补齐到 response/request 共享入口：如果真实 SDK callable 的 `__signature__` 元数据本身抛错或不可检查，runner 会按未知签名处理并继续尝试真实 dump 调用；根层 response parsed body、嵌套 response typed payload、request `json_body`、`query_params` 与 `headers` 的 typed dump 都已覆盖，避免坏 signature metadata 被误报成 transport/config 错。
336. HTTP JSON typed dump-json string wrapper fallback 已一次性补齐到 response/request 共享入口：`model_dump_json()` / `to_json()` / typed `.json()` 返回 `UserString` 这类 string-like wrapper 时会按 UTF-8 JSON 文本处理；根层 response parsed body、嵌套 response typed payload、request `json_body`、`query_params` 与 `headers` 均已覆盖，避免真实 SDK wrapper 被误报成 body/query/header 协议错误。
337. HTTP JSON static diagnostics 的 typed request value 归一化也已补齐：`headers`、`query_params`、`json_body` 的 direct typed value 与 supported runtime template 渲染出的 typed value 会先复用 request runtime 的 JSON-compatible coercion，再进入 object/value、Content-Type/Accept、duplicate query/header 与 reserved runtime template typo 诊断；typed adapter dump 内部的 `settings_*` / `tool_registry_*` typo 不再漏过 preflight，真实可执行 wrapper 也不再被误报成协议错误。
338. HTTP JSON execution summary 的 typed/wrapper 侧路也已补齐：direct typed request mapping、`UserString` response_path 与 root template 渲染出的 typed `result_fields` 会先复用静态归一化，再生成 header/query/body count、response_path 与 result_field_names；通用 template render/missing/reference 扫描也会把 `UserString` 当文本处理，避免 settings/preflight 摘要递归或漏字段。
339. HTTP JSON control field string wrapper 也已补齐：`url`、`method`、`timeout_ms` 的 `UserString` 会按明确 string-like value 进入 validation、runner 与 execution_summary；`method` runtime template 也会在 execution_summary 中按静态上下文渲染，避免 settings/preflight/tool_start 摘要方法与真实执行方法不一致。
340. HTTP JSON execution kind string wrapper 也已补齐：`execution.kind=UserString("http_json")` 会按真实执行器入口枚举进入 validation、runner dispatch 与 execution_summary；普通 object 仍保持缺失/unsupported 语义，避免随意 stringify 坏配置。
341. HTTP JSON runtime semantic meta 的半迁移 summary/diagnostics wrapper 也已补齐：registration 自身携带的 `execution_summary` / `execution_diagnostics` 若使用 `UserString` key/value/list item，会在 `tool_start`、action meta、trace/export 入口统一按 string-like value 归一并脱敏，不再因为 wrapper 丢失摘要/诊断或泄漏敏感 path/key。
342. HTTP JSON runtime semantic kind wrapper 也已补齐：registration 的 `kind` / `runtime_semantic_kind` 支持 `UserString` 窄归一化，noncanonical real tool 仍保留工具名级 trace `semantic_kind`，同时稳定补出 family 级 `semantic_family`，避免 settings/preflight 与运行态 trace 语义分叉。
